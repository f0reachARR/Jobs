# ADR-0007 Quest を別プラグインに分離する

## ステータス

受け入れ

## 背景

「100 体倒したら 5,000G」のようなマイルストーン報酬を Job プラグインで扱う必要がある。
これを実装するには、何らかのクエスト的なフレームワークが要る。

実装場所には三つの選択肢がある。

第一に、Job プラグインに「クエスト機能」をオプションとして組み込む案。
Job プラグインの YAML に `quests:` セクションを足し、進捗管理とリワード支払いも Job プラグイン側で行う。

第二に、Quest プラグインを別実装として作り、Job プラグインはイベントとクエリ API だけを公開する案。
Job プラグインは進捗管理を持たない。
マイルストーンは Quest プラグインの一つの利用例として実装する。

第三に、Job プラグインに簡易クエスト機能を内蔵しつつ、将来別プラグインへ抽出できる API 設計にする案。

第一案は短期的には小さく済むが、Job プラグインの責務が膨らむ。
Daily/Weekly のローテーション、複合 objective、Quest journal UI といった「クエストらしい」要求が出てくると、Job プラグインの肥大化が止まらなくなる。

第三案は妥協案だが、内蔵実装と分離実装の両方を維持することになり、抽出時に互換性問題が出やすい。

## 決定

第二案を採用する。

Job プラグインはマイルストーン機能を持たない。
代わりに、報酬支払いを通知する `JobActionPaidEvent` と、行動ログを集計する `ActionLogQueryService` を公開する。
Quest プラグインはこれらを購読・利用して、進捗管理、達成判定、報酬付与、Journal UI を独自に持つ。

Quest プラグインのスコープには次が入る。

- クエスト定義の読み込み
- 進捗の永続化（独自テーブル）
- ObjectiveType と RewardType の拡張点
- 達成セレモニーの Dialog
- Quest journal の Dialog

## 結果

- Job プラグインの責務が「アクション検知、報酬計算、行動ログ、専業」に絞られる。
- Quest プラグインは Job プラグインがなくとも、`ActionLogQueryService` を公開する代替プラグインがあれば動く。
  逆に Job プラグインは Quest プラグインの存在を前提にしない。
- マイルストーン advancement の同梱データパックは Quest プラグインの責任になる。
  Job プラグインの同梱データパックは Track B 検知用 advancement のみとなる。

## 選択しなかった代替案

第一案は責務肥大の懸念がある。
第三案は二重維持コストが大きく、設計の意図が「分離前提か内蔵前提か」で曖昧になる。
最初から分離するほうが設計の方向性が明確になる。
