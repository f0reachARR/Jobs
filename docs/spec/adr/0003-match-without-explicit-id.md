# ADR-0003 match に明示 ID を持たせず派生キーで識別する

## ステータス

受け入れ

## 背景

`rewards` エントリの識別子をどう持つかには複数の案がある。

第一に、YAML エントリに `id:` フィールドを明示する案。

```yaml
rewards:
  - id: kill_zombie
    on: entity_killed
    entity: minecraft:zombie
    reward: 5
```

第二に、エントリの配列インデックスを内部 ID にする案。

第三に、match 条件から決定的に派生キーを生成する案。

第一案は、行動ログや単調性ペナルティのバケットに人間可読な名前を残せる点が魅力だが、YAML 記述の冗長性を増す。
特に項目数が多くなる採掘や農業では、`id` を毎回書くのが負担になる。
また、`id` を書き換えると過去の行動ログとの紐付けが切れる罠がある。

第二案は、YAML を並び替えるとログとの紐付けが切れる。
順序を変えるたびに集計が壊れるのは運用上耐えがたい。

第三案は、`on:` と match フィールドの内容から自動的にキーを生成する。
例えば `on: entity_killed`、`entity: minecraft:zombie` から `kill:minecraft:zombie` を生成する。
リストやタグは要素を正規化して安定化する。

## 決定

第三案を採用する。

派生キーは `on:` ごとのプレフィックスと match フィールドの値から決定的に生成する。
リスト指定はソートしてから連結し、タグ指定はタグ文字列をそのまま含める。

派生キーは「行動ログの `action_key` カラム」と「単調性ペナルティのバケット識別子」を兼ねる。

## 結果

- YAML エントリに ID を書く必要がない。
- 派生キーは match 条件と一対一対応するため、運営が YAML を見ながらログを検索できる。
- リスト OR とタグ OR の単位が、自動的に単調性ペナルティのバケット単位になる。
  バケットの粒度を選ぶことが、エントリの分け方を選ぶことと同義になる（[ADR-0005](./0005-or-expression-strategy.md) を参照）。
- 派生キーは match 条件の変更で変わる。
  運用開始後にエントリの match を変えると、それまでのログとの紐付けが切れる。
  運用ルールとして「match を変えるなら新規エントリを足し、古いエントリは残す」を推奨する。

## 選択しなかった代替案

明示 ID 案は冗長で、書き換え時の紐付け切れリスクが残る。
配列インデックス案は並び替え耐性がなく、運用に耐えない。
