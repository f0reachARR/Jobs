# ADR-0004 報酬定義は first match wins で評価する

## ステータス

受け入れ

## 背景

1 つの Bukkit イベントが複数の `rewards` エントリにマッチしうる場合、どのエントリを採用するかには方針が必要である。

第一に、最もマッチ条件が specific なエントリを優先する案。
ただし「specificity」をどう定義するかが曖昧で、エンティティ ID 指定とタグ指定の優先関係、複数フィールドの組み合わせの扱いなど、ルールが複雑になる。

第二に、上から順にスキャンして最初にマッチしたエントリを採用する案（first match wins）。
ルールが単純で、YAML を書く運営の予測可能性が高い。

第三に、マッチしたすべてのエントリで報酬を支払う案。
1 イベントで複数の報酬が支払われ、行動ログにも複数行残る。
意図せぬ二重支払いを誘発しやすい。

## 決定

第二案を採用する。

`rewards` 配列を上から下に向かってスキャンし、最初にマッチしたエントリで報酬を確定する。
それ以降のエントリは評価しない。

ただし、`first match wins` ルールには「汎用エントリを上に置くと特化エントリが影に隠れる」落とし穴がある。
これを起動時に検出する shadow チェッカーを内蔵する。
各エントリの match 集合（タグ展開後の集合）を計算し、先行エントリが後続エントリの集合を完全に覆っている場合、警告ログを出す。

## 結果

- ルールが単純で、YAML を書く運営が予測しやすい。
- 1 イベントごとに 1 行のログが書かれる前提が崩れない。
- shadow チェッカーが起動時警告を出すため、汎用エントリと特化エントリの順序ミスを早期に発見できる。

## 選択しなかった代替案

specificity による優先順位は、ルールが定義しづらく、運営が「なぜこのエントリが採用されたのか」を追えなくなる。
複数エントリ全部発火案は、二重支払いの危険性が高く、行動ログの整合性も損なう。
