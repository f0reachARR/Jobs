# ADR-0010 UI は DialogAPI を主とし Inventory GUI は Job 内で使わない

## ステータス

受け入れ

## 背景

Minecraft プラグインの UI 構築には伝統的に Inventory GUI（チェスト型 GUI に擬装したクリック UI）が広く使われてきた。
近年 Paper が DialogAPI を提供し、Vanilla ネイティブのダイアログを使う選択肢が増えた。

Job プラグインで UI が必要な箇所は次の通り。

- 初回ログイン時の専業選択
- 専業変更の確認
- ステータス表示

これらはいずれも「文章を読み、選択肢を選ぶ」体験であり、アイテムスロットへの操作が意味を持たない。

Inventory GUI と DialogAPI のトレードオフを整理する。

Inventory GUI の利点。

- 古くから使われており、サーバ運営とプレイヤーの双方に馴染みがある。
- アイテム表示が中心の体験（ショップ陳列、買い物カゴ）に自然にフィットする。

DialogAPI の利点。

- Vanilla ネイティブで描画され、Minecraft 標準 UI と整合する。
- テキスト入力、ボタン、選択肢を直接的に扱える。
- アイテムスロットを「使っている振りをする」必要がない。

## 決定

Job プラグインの UI は DialogAPI を主に使う。
Inventory GUI のフォールバック実装は持たない。

ただし、プラグイン全体の方針としては「アイテムスロットへの操作が体験の中心になる」場面では Inventory GUI を適切に使う。
具体的には Shop プラグインの陳列と買い物、個人ショップのチェスト枠管理がこれにあたる。
Job プラグイン内ではこの種の場面が存在しないため、Inventory GUI を使わない。

## 結果

- Job プラグインの UI 実装は DialogAPI のみで完結する。
- DialogAPI が experimental ステータスの間は、API 変更追随コストが発生する余地がある。
  Paper 1.21 系の安定化を待ちつつ実装する。
- Inventory GUI 環境を前提とした player 操作（クリック、シフトクリック）の自前ハンドリングが不要になる。

## 選択しなかった代替案

Inventory GUI フォールバック併設案は、二重実装コストと UI 挙動の不整合リスクを生む。
DialogAPI 単独で進められない事情（experimental の不安定さ、致命的バグ）が発生した時点で再検討する。
