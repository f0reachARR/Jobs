# ADR-0018 リポジトリ層をインタフェースで切る

## ステータス

受け入れ

## 背景

Phase 1 の永続化バックエンドは MySQL に決めている（[ADR-0009](./0009-mysql-persistence.md)）。
一方で、次の理由から別方言（特に SQLite）を後付けで差し込む余地は残しておきたい。

- 開発環境で MySQL コンテナを立ち上げず、単一の `.db` ファイルだけで統合テストや動作確認を回したい場面がある。
- 単一 Paper インスタンス・小規模運用のサーバ管理者が MySQL 未導入で試せる経路を潰したくない。
- Phase 2 で複数 Paper インスタンス化した場合でも、開発・staging 環境は SQLite で回して production を MySQL にする分岐は現実的。

方言ごとに DAO を書き分ける戦略はいくつかある。

- **jOOQ・Hibernate 等の方言抽象化ライブラリ**：一つの SQL・DSL から複数方言の SQL を吐かせる。抽象化コストが高く、両方言で意図しないクエリが吐かれる余地がある。[ADR-0009](./0009-mysql-persistence.md) で採用しない方針を確認済み。
- **文字列テンプレートで SQL を共有し、プレースホルダで方言差を吸収**：`BINARY` と `BLOB` のような単純な差は表現できるが、UPSERT 構文やパーティションのような大きな差では破綻する。
- **DAO インタフェースだけ共有し、SQL は方言別に丸ごと書く**：抽象度をメソッドレベルに置き、SQL は共有しない。書き分けコストは払うが、各方言のクエリを最適な形で書ける。

Phase 1 では MySQL 一本で走る以上、SQLite 実装が実際に必要になるかは不確定である。
必要になったときに、MySQL 側のクエリを妥協せずに済む形が望ましい。

## 決定

リレーショナルストアへのアクセスはリポジトリインタフェース越しに限定する。
具象実装は方言別のサブパッケージに置き、SQL・DDL・マイグレーションを共有しない。

Phase 1 で提供する具象は MySQL 実装のみ。
SQLite 実装は Phase 2 の余地として空けておく。

共有するのは次の 3 種類のみとする。

- リポジトリインタフェースのメソッドシグネチャ
- 引数・戻り値に使う DTO 型（`PlayerJobRow`, `ActionLogRow`, `ActionFilter`, `TimeRange` 等）
- スレッドモデル・保持期間などのふるまいに関する規約（[05-persistence.md](../05-persistence.md) に記載）

SQL 文字列、DDL、UPSERT 構文、パーティション、接続プール設定、マイグレーションスクリプトは方言別に完全に独立させる。

方言抽象化ライブラリ（jOOQ、Hibernate）は引き続き入れない。

## 結果

- MySQL 実装は MySQL に最適化した SQL（`BINARY(16)`、`ON DUPLICATE KEY UPDATE`、`RANGE PARTITION`）を、SQLite の都合で妥協せずに書ける。
- SQLite 実装を追加するときも、`BLOB`、`INSERT ... ON CONFLICT DO UPDATE`、単一 write 接続 + WAL のように SQLite に最適な形で書ける。
- SQLite 実装追加のコストは、テーブル 3〜4 種類ぶんの CRUD を丸ごと二重に書く程度に留まる。
  インタフェースが安定していれば呼び出し側の書き換えは不要。
- インタフェースが痩せているぶん、方言抽象化に由来するバグ（意図しない SQL が吐かれる、方言差で挙動が変わる）は原理的に起きない。
- リポジトリインタフェースの命名は SQL に寄せないことに注意する。
  `selectByPlayer` ではなく `findCurrent`、`upsertDailyTotal` ではなく `addBatch` のように、ドメイン操作として命名する。

## 選択しなかった代替案

- jOOQ・Hibernate による方言抽象化：[ADR-0009](./0009-mysql-persistence.md) の方針と重複するため採用しない。
- 文字列テンプレート + 方言別プレースホルダ：UPSERT やパーティションのような大きな差を表現しきれず、途中で破綻する。中途半端な抽象化は、後で SQLite を追加する側にも MySQL を書く側にも制約を残す。
- DAO も方言ごと独立でインタフェースを切らない：MySQL 実装しか呼ばれない状態から SQLite に切り替える経路が、呼び出し側の書き換えを要する。インタフェースだけは最初から切っておく。
