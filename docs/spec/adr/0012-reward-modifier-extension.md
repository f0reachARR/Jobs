# ADR-0012 報酬パイプラインの拡張点を Modifier と Splitter で公開する

## ステータス

受け入れ

## 背景

イベント固有のルールには「報酬倍率」「報酬の一部を別口座へ流す」の二系統がある。

倍率の例。

- 期間ボーナス（参加日からの経過時間に応じた ×1.3〜×4.0）
- 挽回イベント倍率（Day 4 大鉱脈で採掘 ×2 など）
- ツール強化アイテム保有時の永続倍率（×1.2）

分配の例。

- 社員拠出 20%（10% を会社財布、10% を自社株自動付与原資）
- 救済ローン天引き 20%

両者は性質が異なる。
倍率は「報酬額を変える」処理であり、入金額は変わるが分配先は単一である。
分配は「報酬額の一部を別口座に振り分ける」処理であり、プレイヤー入金額が減る代わりに別の口座が増える。

これらを単一の拡張点（汎用 Modifier）でまとめて扱う案もあるが、報酬の意味論が違うため混ぜると次の不都合が出る。

- 行動ログに記録すべき「Modifier 適用後の最終報酬」と「実際の入金額」が同じものになる。
  社員拠出を Modifier で実装すると、ログの `final_reward` がプレイヤー入金額になり、業績指標として「報酬」を集計したいときの値が分かりにくくなる。
- 倍率の順序と分配の順序を別々に管理したいケースに対応しづらい。

## 決定

拡張点を二系統に分ける。

- **JobRewardModifier**：報酬額を変える拡張点。優先度順に適用する。`final_reward` を更新する。
- **JobRewardSplitter**：報酬の一部を別口座に分配する拡張点。`final_reward` を変えず、分配後にプレイヤーへ送る額（`net_paid`）を確定する。

行動ログは `base_reward`、`final_reward`、`net_paid` の三つを記録できる構造にする（[05-persistence.md](../05-persistence.md) では `base_reward` と `final_reward` の二つを記録し、`net_paid` は別系統のログに残す）。

両拡張点は `LifecycleEvents.JOB_PLUGIN_READY` で登録する。
登録 API は ID 付きで何度でも呼べる。

## 結果

- 倍率系と分配系の責務が明確に分離される。
- 行動ログの `final_reward` は「業績評価のための報酬値」として一貫した意味を持つ。
  業績指標バッチ（株プラグイン）は `final_reward` を集計すれば、プレイヤーごとの活動量を正しく比較できる。
- Splitter の分配先と分配額の組み合わせは Splitter 実装の内部に閉じる。
  会社財布、自社株原資、ローン返済口座のような口座種別は Economy プラグインの口座 API に依存する。

## 選択しなかった代替案

単一拡張点案は、報酬の意味論を混ぜる結果になり、業績評価の指標が曖昧になる。
拡張点を 3 つ以上に分ける案（pre-Modifier、Modifier、post-Modifier、Splitter のような細分化）は、現時点の要件で必要性がなく、過剰設計になる。
