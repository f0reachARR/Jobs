# ADR-0009 永続化を MySQL とする

## ステータス

受け入れ

## 背景

行動ログ、専業履歴、日次キャップ集計の永続化バックエンドの候補は、SQLite、MySQL、PostgreSQL、Redis、ファイルベース（YAML や JSON）がある。

要件として次が見えている。

- 1 アクションあたり 1 行の追記書き込み。
  100 名規模で 1 日 100,000 行クラス。
- プレイヤーごと・ジョブごと・時間範囲の集計クエリ。
  業績指標バッチ（株プラグイン）が毎時叩く。
- プロキシ配下の複数 Paper インスタンスから同時アクセスする余地。

ファイルベースは集計クエリで破綻する。
Redis は永続性と容量の双方で不向き。
SQLite は単一プロセス前提が強く、複数 Paper 想定と相性が悪い。
MySQL と PostgreSQL のいずれかが現実的な選択肢になる。

MySQL は JOIN や複雑クエリで PostgreSQL に劣る局面もあるが、本プラグインの集計クエリは単純な GROUP BY が中心で、両者の差は出にくい。
運用面では、Minecraft サーバ系のホスティングで MySQL がデフォルトで用意されているケースが多く、サーバ管理者にとって馴染みが深い。

## 決定

Phase 1 の永続化バックエンドは MySQL とする。
接続プールは HikariCP を使う。
スキーマは [05-persistence.md](../05-persistence.md) で定義する。

DAO は方言別に丸ごと書く方針を取り、SQL 共有はしない。
SQLite など別方言の実装を後付けする余地は、リポジトリインタフェース越しに残す（[ADR-0018](./0018-repository-interface.md)）。
Phase 1 で提供する具象実装は MySQL 版のみ。

## 結果

- 複数 Paper インスタンス構成への拡張余地が残る。
- サーバ運営は MySQL（または互換の MariaDB）の用意が必要になる。
  共有ホスティングの多くで用意されている範囲。
- Phase 1 の DAO 実装は MySQL 方言に直接依存して書く。
  方言抽象化のためのライブラリ（jOOQ、Hibernate）は使わず、JDBC + 手書き SQL とする。
- 他方言の実装を追加するときも SQL・DDL・マイグレーションを共有しない（[ADR-0018](./0018-repository-interface.md)）。

## 選択しなかった代替案

SQLite はシングルプロセス前提が強く、運営拡張時（プロキシ配下の複数 Paper インスタンス構成）のボトルネックになる。
Phase 1 の第一実装には採用しない。
一方で、開発環境や単一インスタンスの小規模運用の余地として、リポジトリ層のインタフェース越しに後付けする道は残す（[ADR-0018](./0018-repository-interface.md)）。

PostgreSQL も技術的には妥当だが、Minecraft サーバ系の運用慣習との合致度で MySQL を選ぶ。
将来 PostgreSQL サポートを追加する場合は、SQLite と同じくリポジトリ層の別実装として追加する。
