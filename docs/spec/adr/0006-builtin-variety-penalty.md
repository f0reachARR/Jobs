# ADR-0006 単調性ペナルティを Job プラグインに内蔵する

## ステータス

受け入れ

## 背景

単調性ペナルティ（同じアイテムばかり扱うと報酬倍率が下がるルール）の実装場所には選択肢がある。

第一に、Job プラグイン本体に内蔵する案。

第二に、拡張 Modifier として外部プラグインで実装する案。
Job プラグインは行動ログ集計のクエリ API を公開し、外部プラグインがそれを叩いて倍率を計算する。

第二案は責務分離の点では綺麗だが、単調性ペナルティが「直近 N アクション」を毎回参照する性質を考えると、外部プラグインから MySQL クエリを毎報酬ごとに飛ばすコストが大きい。
in-memory ring buffer での実装が前提になるが、これは Job プラグインの行動ログ機構と密結合する。
分離すると同等の機構を二重に持つことになる。

加えて、単調性ペナルティは「直近の活動パターン」という Job プラグインがすでに保持しているデータに対する 1 つの関数であり、概念的にも Job プラグインの内側にある。

## 決定

第一案を採用する。

単調性ペナルティは Job プラグインに内蔵する。
ジョブごとの YAML で `variety_penalty` セクションを有効化することで使う。
カーブと窓幅は YAML で設定する。

カテゴリ分け（鉱石グループ、魚種グループ）はサポートしない。
バケットの単位は派生キーであり、バケットをまとめたい場合は match のリスト OR またはタグ OR でエントリを束ねる（[ADR-0005](./0005-or-expression-strategy.md) を参照）。

## 結果

- 単調性ペナルティの実装は Job プラグイン内の `variety_penalty` 計算器に集約される。
  in-memory ring buffer を効率良く使える。
- カテゴリ分けがないため、グループ化したい場合の意思決定が YAML エントリの分け方に集約される。
  運営が「どの粒度で多様性を評価したいか」を YAML 構造で表現することになる。
- 拡張 Modifier として外部プラグインから報酬倍率を上書きしたい場合（例：特定のイベントだけペナルティを免除する）は、内蔵 Modifier の後段で拡張 Modifier を適用する報酬パイプラインの順序により、外部から打ち消せる。

## 選択しなかった代替案

外部プラグイン実装案は責務分離としては綺麗だが、in-memory ring buffer を外部に持ち出すか、毎回 MySQL を叩くかの 2 択になり、どちらも本機構の性質と合わない。
カテゴリ分けサポート案は、追加の YAML フィールドと運用ルールの複雑化を招き、その対価として得られる表現力は OR の使い分けで十分代替できる。
