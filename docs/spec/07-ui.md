# UI（BedrockDialog）

Job プラグインの UI は [BedrockDialog](https://github.com/f0reachARR/BedrockDialog) の Unified API を通して構築する（[ADR-0010](./adr/0010-dialog-api-over-inventory-gui.md)、[ADR-0014](./adr/0014-bedrock-dialog.md)）。
BedrockDialog は Java Edition プレイヤーには Paper DialogAPI を、Bedrock Edition プレイヤーには Geyser フォームを、プレイヤーごとに自動で切り替えて描画する。
Job プラグイン側は Java/Bedrock を意識せず、`ConfirmDialog` / `NoticeDialog` / `MultiButtonDialog` / `InputDialog` の型で UI を組む。

Inventory GUI は Job プラグイン内では使わない。
Inventory GUI が体験の中心になるのは「アイテムスロットへの操作そのものが意味を持つ」場面であり、Job のインタラクションはそうではない。

## Dialog を使う 4 箇所

### 1. 専業一覧ダイアログ（`SpecialtyListDialog`）

`MultiButtonDialog` を使う。以下のいずれかの契機で開く。

- 初回ログイン時（`PlayerJoinEvent` のうち `player_job` に行が存在しないプレイヤー。`config.yml` の `specialty_mode.show_select_dialog_on_join: true` のときのみ）。
- `/jobs select` コマンド（未選択なら `SELECT` モード、選択済みかつ変更可能なら `CHANGE` モード、cooldown 中なら第 3 節の `SpecialtyCooldownDialog` に分岐）。

表示内容。

- 5 職業のアイコン（YAML の `icon` フィールドから）。Bedrock 側は Dialog アイコンを持たないため無視される。
- 表示名、簡単な説明、代表的な報酬例（ボタン本文または上部の body に）。
- `CHANGE` モードでは現在の専業を除外する。
- 職業ごとに「この職業を選ぶ」／「この職業に変更する」ボタン。

`config.yml` の `specialty_mode.disclose_before_select: true`（default）のとき、ボタン押下で即確定せず、まず第 2 節の `JobConditionsDialog` を開き、そこで再度「この職業を選ぶ／変更する」を選ばせる。
`false` のときは従来通り 1 クリック確定。

ダイアログを閉じた場合のフォールバックは設けない。
専業未選択のプレイヤーは、すべてのアクションで報酬を得られず、行動ログにも残らない。
`/jobs select` コマンドを別途用意し、いつでも一覧ダイアログを再表示できるようにする。

### 2. 職業条件の開示ダイアログ（`JobConditionsDialog`）

`MultiButtonDialog` を使う。単職の条件をひと画面にまとめる。以下のモードで挙動が変わる。

- `PREVIEW_FOR_SELECT`：一覧から遷移。ボタン `[この職業を選ぶ] [一覧に戻る]`。確定で `SpecialtyService#select`。
- `PREVIEW_FOR_CHANGE`：変更フロー中。ボタン `[この職業に変更する] [一覧に戻る]`。確定で `SpecialtyService#change`。
- `READ_ONLY`：`/jobs info` 経由。ボタン `[閉じる]`（引数なし経路から来た場合は `[一覧に戻る]`）。

body に並べる内容。

- 職業表示名と、YAML の `description`（省略時は行を出さない）。
- **報酬対象アクション**：`rewards[]` を `ActionType` でグルーピング。`entity_killed`/`block_broken`/`item_smelted` … ごとに「討伐: minecraft:zombie」「採掘: #minecraft:ores（タグ）」のように list / tag / single を人語で列挙する。1 職業あたりの上限件数を実装側で持ち、超過分は「他 N 件」と表示する。
- **報酬額**：`config.yml` の `specialty_mode.disclose_reward_amount: true`（default）のとき、`RewardAmount.Fixed` は値、`Range` は "min〜max" を対象に併記する。`false` のとき額は伏せ、対象のみ列挙する。数値から表示文字列への変換は Vault Economy の `format(double)` に委譲し、通貨単位や小数桁は Economy provider 側に集約する（プラグイン内で独自に丸めや通貨名を持たない）。
- **rare 表示**：rare を含むエントリには「レア報酬あり」の一言を末尾に添える。`chance` と `reward` の数値は `disclose_reward_amount` に関わらず出さない（発動時の `announce` で初めて可視化される演出を尊重）。
- **単調性ペナルティ**：有効時は `disclosed_message` のみを表示する。数値情報（`curve` の閾値、`window` 件数）は `hide_numbers` と一貫させ、事前開示でも出さない。無効時は「なし」。`disclosed_message` 未設定のときは「有効」とだけ表示する。
- **自動化対策**：`AntiAutomationConfig.merge` で config.yml の default と per-job override を合成したうえで、有効化されているチェックだけを列挙する。文言は既存の `notify.anti_automation.*` を流用しつつ、`dialog.info.anti_automation.*` に別途 key を用意する（発動時通知と事前開示で trafficking を切り分けたい場合の余地）。

「一覧に戻る」ボタンは `closeDialog` に依存せず、`SpecialtyListDialog` を明示的に開き直す形で実装する（[ADR-0014](./adr/0014-bedrock-dialog.md)）。

### 3. 変更クールダウンダイアログ（`SpecialtyCooldownDialog`）

`/jobs select` を選択済みプレイヤーが実行し、`nextAvailableAt` が未経過のときに開く。
`NoticeDialog` を使い、以下を表示する。

- 現在の専業。
- 次回変更可能な時刻と、あと残り時間。
- 閉じるボタンのみ。

このダイアログからは職業選択の導線を出さない（一覧を開かせない）。cooldown 中の再選択は仕様上ブロックする。

### 4. `/jobs status` のステータス表示

Dialog は使わず、チャット欄に MiniMessage メッセージとして出力する。
表示情報が「読むだけ」で相互作用が発生せず、Dialog を開くコストに見合わないため。

表示内容。

- 現在の専業と、その累計獲得額。
- 本日の獲得額と、`daily_cap.amount` に対する進捗。
- 単調性ペナルティの状態。`hide_numbers: true` のとき、`disclosed_message`（「同じものばかり扱うと効率が落ちる」）のみを表示し、比率や倍率は出さない。発動していないときは何も表示しない。
- 次の専業変更可能時刻。

クエストの進捗は表示しない。
クエスト関連の表示は Quest プラグイン側のダイアログで担う（[ADR-0007](./adr/0007-quest-as-separate-plugin.md)）。

## `/jobs select` の分岐

`/jobs select` は「専業を決める／見直す」動線を 1 コマンドに集約する。
旧 `/jobs change` は削除し `/jobs select` に統合する。

- 現専業なし → `SpecialtyListDialog(Mode.SELECT)` を開く。
- 現専業あり、`nextAvailableAt` 未経過 → `SpecialtyCooldownDialog` を開く。
- 現専業あり、変更可能 → `SpecialtyListDialog(Mode.CHANGE)` を開く。

`/jobs info [<job-id>]` は純閲覧経路として別に用意する。引数なしなら `SpecialtyListDialog(Mode.INFO)` を開き、各ボタンから `JobConditionsDialog(READ_ONLY)` を「一覧に戻る」付きで表示する。引数ありなら該当ジョブを直接 `JobConditionsDialog(READ_ONLY)` で開き（一覧経路に戻る必要が無いため「閉じる」のみ）、選択・cooldown 状態に関係なく実行できる。

## Dialog リソースの配置

Dialog は Java コード上で BedrockDialog の Unified API（`ConfirmDialog.builder()` などの `UnifiedDialog`）を使ってプログラム的に構築し、`BedrockDialog.get().show(player, dialog)` で表示する。
Vanilla データパック（`data/jobs/dialog/` の JSON）は使わない。
データパック JSON は locale 別の差し替えができず、多言語対応の要件と両立しないため（[ADR-0013](./adr/0013-i18n-yaml-minimessage.md)）。

動的な情報（プレイヤー名、報酬額、クールダウン時刻）は MiniMessage のプレースホルダとして翻訳文字列に埋め込み、Dialog 構築時に `TagResolver` で解決してから `Component` として BedrockDialog に渡す。

BedrockDialog 依存の宣言方法（`paper-plugin.yml` の `dependencies.server`）、および `BedrockDialog.init` / `BedrockDialog.reset` の呼び出しは [BedrockDialog API](https://github.com/f0reachARR/BedrockDialog/blob/main/API.md) に従う。

## Bedrock Edition 対応の注意点

BedrockDialog は Java/Bedrock で描画バックエンドが異なるため、Job プラグイン側のコードは次の制約を前提にする（[ADR-0014](./adr/0014-bedrock-dialog.md)）。

- **MiniMessage 装飾は Bedrock 側で strip される**。装飾に依存した情報表現（色分けだけで「変更不可」を示す等）を作らず、文言そのもので伝わる書き方にする。
- **コールバックはメインスレッドから呼ばれるとは限らない**。Bedrock 側ではネットワークスレッドから呼ばれる可能性がある。Bukkit API を触る処理は `Bukkit.getScheduler().runTask(plugin, ...)` に載せる。専業書き込み、`JobSpecialtyChangedEvent` 発火、行動ログ書き込みは常にメインスレッドにディスパッチする。
- **プログラム的な `closeDialog` は Bedrock では Floodgate が必要**。Job プラグイン内では、明示 close に依存する動線を作らない（プレイヤーが自分で閉じる、または選択で自然に閉じるフローに揃える）。`JobConditionsDialog` の「一覧に戻る」は新しい dialog を開くことで代替する。
- **`InputDialog` のスライダ step は Bedrock 側で整数丸め**。Job プラグイン内で `InputDialog` を使う場面が発生した場合、step を整数で設計する。
- **on-close コールバックは Bedrock 側で発火しない**。close 契機でのステート遷移を組まない（初回選択の未確定状態は、明示ボタン以外の経路で確定させない）。
- **body の描画長には限りがある**。`JobConditionsDialog` の報酬一覧は上限件数で truncate し、超過分は "他 N 件" と示す。

## 多言語対応

UI 文言（Dialog のタイトル、本文、ボタンラベル、`/jobs` 系コマンドのシステムメッセージ）は `lang/<locale>.yml` に [MiniMessage](https://docs.advntr.dev/minimessage/format.html) 形式で定義する（[ADR-0013](./adr/0013-i18n-yaml-minimessage.md)）。

- ファイル名は Minecraft のロケール ID に合わせる（`ja_jp.yml`、`en_us.yml`、`ko_kr.yml` 等）。
- 値は MiniMessage 形式。装飾タグ（`<red>`、`<hover>`）と、`<player>`・`<amount>`・`<cooldown>` のような動的プレースホルダを併用できる。
- キーは Dialog 単位で階層化する（`dialog.specialty.title`、`dialog.info.header.rewards`、`dialog.info.button.select` 等）。
- プレイヤーの locale は `Player#locale()` で解決する。対応するロケールファイルが無ければ `ja_jp` にフォールバックする。`ja_jp` にもキーが無い場合は起動時警告を出し、実行時はキー名をそのまま返す。
- 起動時に全ロケールファイルを読み込み、キー集合の差分（あるロケールにだけ存在するキー、`ja_jp` にあって他ロケールに欠けているキー）を警告として出す。
- Bedrock Edition プレイヤーには BedrockDialog が MiniMessage 装飾を strip して表示する。翻訳文字列は装飾なしでも意味が通るよう書く（色分けや太字だけで意味を持たせない）。

ジョブ YAML の `display_name`、`description`、`variety_penalty.disclosed_message`、`rewards[].rare.announce` はイベント固有のジョブ定義に強く紐づくため翻訳ファイルには置かず、値そのものを MiniMessage として解釈する。
イベント側で locale 別に切り分けたい場合は、それぞれのジョブ YAML を locale 別に用意するのではなく、イベント側プラグインが翻訳ファイル側のキー参照を差し込む拡張点で対応する（詳細は Job プラグイン内では規定しない）。
